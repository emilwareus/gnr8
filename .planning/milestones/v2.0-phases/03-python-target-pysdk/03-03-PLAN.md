<objective>
Write `crates/gnr8-core/tests/pysdk_compile.rs` — the hermetic generate-and-run acceptance test, the
load-bearing proof of the phase and the twin of `tests/sdk_compile.rs`. It builds the IR from
`fixtures/fastapi-bookstore` (via the existing Phase-2 `pyextract` path that `build_graph` already routes
to), generates the Python SDK, writes it into a unique stdlib temp dir, then runs three gates against
`python3`: (a) `py_compile` of every file (syntax), (b) `import <pkg>` (executes class bodies — catches
the dataclass field-order `TypeError` and bad 3.9 annotations), (c) a program-written driver script that
spawns a stdlib `http.server` on ephemeral port 0, injects an `OpenerDirector` into the generated
`Client`, and asserts a 2xx `@dataclass` round-trip plus a 4xx → typed `ApiError`. Because `python3` is
present and `go` is absent in this sandbox, THIS is the test that actually runs (the Go SDK tests skip).

Purpose: PYSDK-02 — the generated SDK imports, "type-checks" (≡ compiles + imports + round-trips under the
stdlib-only constraint), and round-trips against the FastAPI fixture with no third-party HTTP deps.
Output: a passing hermetic integration test that exercises the full generate→write→compile→import→round-trip
path, skipping cleanly if `python3` is absent.
</objective>

The FastAPI fixture's actual IR shape (from the committed graph snapshot — drive the driver assertions off these):

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Rust test harness → `python3` subprocess | The harness spawns `python3`; an injected/shell-interpolated argv would be command injection. |
| program-generated driver/script → temp filesystem | The driver `.py` is written under a temp dir; a predictable/collidable path is a hygiene risk. |
| "hermetic" test → network | The round-trip must stay on localhost with no package fetch (the Python analog of Go's `GOPROXY=off`). |
| generated SDK content → supply-chain claim | PYSDK-01's "dependency-free" is a property of the emitted code that must be asserted, not assumed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Tampering / EoP | `run_python` spawning python3 | mitigate | Discrete `Command::args`, `current_dir`, NO shell string; the driver is written to a FILE and run by path, never `-c "<interpolated data>"`. |
| T-03-03-02 | Tampering / DoS | temp-dir predictability/collision | mitigate | `unique_temp_dir`: `std::env::temp_dir()` + PID + nanos, no user-supplied component (copied verbatim); best-effort cleanup. |
| T-03-03-03 | DoS / hermeticity | test http.server port race / hang | mitigate | Bind `("127.0.0.1", 0)` (ephemeral port), serve in a daemon thread, `shutdown()`/`server_close()` in `finally` (Pitfall 5). |
| T-03-03-04 | Tampering (supply chain) | generated SDK third-party deps | mitigate | The test greps every written `.py` and asserts NO `requests`/`httpx`/`pydantic`; the driver+backend use only Python stdlib (no pip install, no FastAPI run) — rule 2. |
| T-03-03-05 | Robustness | subprocess non-zero exit | mitigate | `run_python` maps non-zero exit to a captured-stderr `CoreError`, spawn failure to `PythonToolchainMissing`; no `unwrap`/`expect` on the subprocess Result (RUST-04). |
| T-03-03-SC | Tampering | npm/pip/cargo installs | mitigate | None — the test installs nothing (`python3` + stdlib already present). Any `pip install` is a defect (rule 2). |
</threat_model>
