<objective>
Implement loop-safe watch mode with debounce and latency reporting (WATCH-01, WATCH-02, WATCH-03). The
`notify-debouncer-full` shell lives in the binary (D-06/D-09); the testable "should this event trigger a
regeneration?" decision is a PURE function unit-tested without a real watcher (RESEARCH Pattern 4, Pitfall 3).
The watch-triggered regeneration reuses 04-02's `regenerate` so no-op filtering (WATCH-01) applies automatically.

Purpose: deliver the third headline guarantee — **watch doesn't loop**. gnr8's own writes to the configured
output paths are filtered out of the event stream (two-layer defense: watch source dirs only + drop
output-path events), so regeneration never re-triggers itself (RESEARCH Pitfall 1). Latency reporting
(cold / warm-no-op / single-file-edit) feeds Phase 5's benchmark evidence.

Output: the `watch` module (debouncer shell + pure event filter + `Instant` latency), the wired `gnr8 watch`
CLI arm with Ctrl-C handling, a timing-tolerant smoke test, unit tests for the pure filter, and the new
`lifecycle`/`watch_smoke` tests added to the blocking make + CI gates.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user filesystem events → watch loop | `notify` reports ALL changes under the watched tree, including gnr8's own output writes |
| watched source dir → regeneration trigger | arbitrary files may change; only supported `*.go` source edits should trigger work |
| Ctrl-C / OS signal → process lifecycle | the long-running watch process must shut down cleanly without orphaning the watcher |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03-01 | Denial of Service | watch self-trigger loop (FSEvents reports gnr8's own writes) | mitigate | two-layer defense: watch SOURCE dirs only (never output dirs) + pure `is_trigger_path` drops any event under an output path / in the manifest; `notify-debouncer-full` coalesces bursts (RESEARCH Pitfall 1) — a correctness AND availability control, unit-tested without a watcher |
| T-04-03-02 | Denial of Service | duplicate regenerations from atomic-save editors (create+rename+modify) | mitigate | `notify-debouncer-full` 0.7.0 merges rename pairs + dedups create/modify within a ≥150ms window (200ms default); smoke test asserts exactly one regen per single save (RESEARCH Pitfall 2) |
| T-04-03-03 | Denial of Service | a `notify` error or a panicking unwrap kills the watch loop | mitigate | callback `Err(errors)` is logged and the loop continues; every fallible call flows through `?`/log-and-continue — no `unwrap`/`expect`/`panic` in the loop (RUST-04, clippy -D warnings) |
| T-04-03-04 | Tampering / Elevation | regeneration inside watch writes to a config output path | mitigate | watch reuses 04-02's `regenerate` → `apply_writes`, which already validates output paths against the project root and protects user-edited files (inherits T-04-02-01/02) |
| T-04-03-SC | Tampering | notify-debouncer-full / notify install (supply chain) | accept | RESEARCH Package Legitimacy Audit: notify-rs org, 11 yrs / 32.3M (notify) + 2.9M (debouncer-full) downloads, source-backed, slopcheck [OK]; pinned to stable 0.7.0 / notify ^8.2.0 (NOT the 0.8.0-rc/9.0.0-rc candidates); no [ASSUMED]/[SUS]/[SLOP] → no blocking checkpoint required |
</threat_model>
