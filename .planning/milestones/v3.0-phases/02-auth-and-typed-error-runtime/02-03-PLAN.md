# Phase 2 Plan 3: Typed SDK Error Runtime

## Goal

Make generated Go, Python, and TypeScript SDKs expose a production-useful base error object for every
non-2xx response, including status, headers/request ID, raw body, parsed JSON body, and a decoded
declared error schema where the operation documents one.

## Requirements Covered

- ERR-01: Exported base SDK error type per target language.
- ERR-02: Error objects expose status, headers/request ID, raw body, and parsed JSON body.
- ERR-03: Error objects expose decoded declared error response schema when present.
- ERR-04: Explicit status errors are resolved deterministically before fallback behavior for currently
  representable graph responses.
- ERR-05: Compile/runtime smoke tests cover Go, Python, and TypeScript behavior where local toolchains
  are available.

## Implementation

1. Add a shared SDK error response resolver for operation non-2xx JSON body models.
2. Extend Go `APIError` with headers, request ID, raw body, parsed JSON body, and typed `Body`; read
   raw response bytes once and decode explicit status bodies by model.
3. Extend Python `ApiError` and `_raise` with headers, request ID, raw body, parsed JSON body, and
   decoded model `body`; import error models referenced by client methods.
4. Extend TypeScript `ApiError` and non-2xx dispatch with headers, request ID, raw text body, parsed
   JSON body, and model-cast `body`.
5. Add focused generator assertions and runtime smoke tests.

Status: passed on 2026-07-09.

## Deferred

OpenAPI `default` / status range precedence remains limited by the current graph importer, which
rejects `default` responses before SDK generation. This plan preserves deterministic explicit-status
behavior and leaves importer support for future graph response-shape work.
