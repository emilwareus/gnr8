# Phase 2 Plan 1: Query API-Key Auth

## Goal

Add `apiKey` security schemes in query parameters across OpenAPI and generated Go/Python/TypeScript
SDKs, while preserving existing header API-key behavior and public-operation overrides.

## Requirements Covered

- AUTH-01: Graph security declarations lower consistently into OpenAPI and SDKs.
- AUTH-02: SDK consumers can configure API-key auth once at client construction.
- AUTH-03: Explicit public operations do not send inherited credentials.
- AUTH-04: Runtime smoke tests verify outgoing auth query params where practical.
- AUTH-05: Unsupported SDK auth schemes still produce clear diagnostics.

## Implementation

1. Add an `ApplySecurity::api_key_query(id, param_name)` transform constructor.
2. Extend OpenAPI lowering/writers to support `apiKey` schemes with `in: query`.
3. Replace header-only SDK auth resolution with operation-scoped API-key schemes carrying `header` or
   `query` location.
4. Emit query API keys in Go, Python, and TypeScript operation dispatch without changing existing
   header-only outputs.
5. Add targeted unit/runtime tests for query auth and inherited/public-operation behavior.

## Verification

- `cargo test -p gnr8 sdk::emit_common`
- `cargo test -p gnr8 lower::`
- `cargo test -p gnr8 --test sdk_compile`
- `cargo test -p gnr8 --test pysdk_compile`
- `cargo test -p gnr8 --test tssdk_compile`

## Deferred

Bearer/basic auth and richer typed error payloads remain later Phase 2 plans.
